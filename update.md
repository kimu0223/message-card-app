## Stripe 日本向けセキュリティ対策（決済導入前に必須）

日本ではクレジット取引セキュリティ対策協議会が定める「セキュリティ・チェックリスト」に基づく
**セキュリティ対策措置状況申告書**の提出が、クレジットカード決済を行うすべての事業者に義務付けられている。

### 決済導入前に実施すること

#### 1. カード情報の非保持化（最重要）
- Stripe Elements（またはStripe.js）を使ってカード情報入力フォームを実装する
- カード番号・有効期限・CVCが自社サーバーを**一切経由しない**構成にする
- Stripeを正しく使えばこれは自動的に達成されるが、フォームをカスタム実装しないこと
- 確認：Route Handlerでカード番号を受け取るコードがないか検索する

#### 2. EMV 3Dセキュア（3DS2）の導入
- Stripe Dashboardの「Radar」設定で3Dセキュアを有効化する
- `PaymentIntent` 作成時に `payment_method_options.card.request_three_d_secure: 'automatic'` を設定
- または Stripe Radarのルールで「3DS認証が必要な場合はブロック」を設定
- 参考：`src/app/api/stripe/` のRoute Handlerを更新する

#### 3. 不正利用対策（Stripe Radar）
- Stripe Dashboardで Radar を有効化する
- 基本的なルール（海外カード・高額取引のレビュー等）を設定する

#### 4. HTTPS対応
- Vercelデプロイであれば自動でHTTPS対応済み
- カスタムドメイン設定後もHTTPS証明書を確認する

#### 5. セキュリティ対策措置状況申告書の提出
- Stripeのアカウントダッシュボードから申告書を提出できる場合がある
- または加盟店契約時にStripeから案内が来る
- 対応時期：本番リリース前に提出が必要

### 対応状況チェックリスト
- [ ] Stripe Elementsでカード入力フォーム実装済み（カード情報が自社サーバーを通らない）
- [ ] EMV 3Dセキュアの設定完了（Stripe Dashboard + コード両方）
- [ ] Stripe Radar有効化・ルール設定完了
- [ ] 本番環境がHTTPS対応済み
- [ ] セキュリティ対策措置状況申告書を提出済み
